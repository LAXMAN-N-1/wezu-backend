## Production Bugs Found & Fixed

### 1. BatteryStatus Enum Mismatch (Critical)

| Component | Values |
|---|---|
| `BatteryStatus` enum (model) | `available`, `rented`, `maintenance`, `charging`, `retired` (5) |
| `VALID_BATTERY_STATUSES` (service) | `available`, `deployed`, `charging`, `faulty`, `in_transit`, `maintenance`, `new`, `ready`, `reserved`, `retired` (10) |

**Impact**: `apply_battery_transition()` writes `"deployed"` to `batteries.status`.
PostgreSQL stores this in a VARCHAR-backed enum column so it round-trips silently,
but SQLAlchemy's Python-level enum validator rejects it on read-back тАФ and the
ORM model is the canonical schema definition.

**Fix**: Added 6 missing members to `BatteryStatus` in `app/models/battery.py`:
`DEPLOYED`, `RESERVED`, `IN_TRANSIT`, `FAULTY`, `NEW`, `READY`.

### 2. LocationType Enum Mismatch (Critical)

| Component | Values |
|---|---|
| `LocationType` enum (model) | `station`, `warehouse`, `service_center`, `recycling` (4) |
| `VALID_LOCATION_TYPES` (service) | `warehouse`, `station`, `customer`, `transit`, `shelf` (5) |

**Impact**: Same issue тАФ service writes `"customer"`, `"transit"`, `"shelf"` to
`batteries.location_type`, which the ORM enum rejects.

**Fix**: Added 3 missing members to `LocationType`: `CUSTOMER`, `TRANSIT`, `SHELF`.

### 3. RentalService.return_battery тАФ Wrong Field Name (High)

**Bug**: `rental_service.py:483` set `rental.drop_station_id = station_id`, but
the `Rental` model only has `end_station_id`. SQLModel raises
`ValueError: "Rental" object has no field "drop_station_id"`.

**Fix**: Changed to `rental.end_station_id = station_id`.

### 4. Missing Battery Lifecycle Fields (Medium)

`battery_consistency.py` references `battery.retirement_date`,
`battery.decommissioned_at`, `battery.decommissioned_by`, and
`battery.decommission_reason` тАФ none of which existed on the Battery model.

**Fix**: Added all 4 fields to `Battery` model with nullable defaults.

## Pre-Existing Issues Found (Not Fixed тАФ Documented for P5)

### 5. RentalService.initiate_rental тАФ Pervasive Field Mismatches

`initiate_rental()` references these fields that **do not exist** on the `Rental` model:

| Service Field | Model Equivalent | Status |
|---|---|---|
| `rental.pickup_station_id` | `rental.start_station_id` | тЭМ Wrong name |
| `rental.rental_duration_days` | *(missing)* | тЭМ Not in model |
| `rental.daily_rate` | *(missing)* | тЭМ Not in model |
| `rental.damage_deposit` | `rental.security_deposit` | тЭМ Wrong name |
| `rental.discount_amount` | *(missing)* | тЭМ Not in model |
| `rental.total_price` | `rental.total_amount` | тЭМ Wrong name |
| `rental.promo_code_id` | *(missing)* | тЭМ Not in model |

Similarly, `rental_in.pickup_station_id` is used throughout the service, but
`RentalCreate` schema has `start_station_id`.

The entire `initiate_rental` тЖТ `create_rental_payment_order` тЖТ `confirm_rental_verified`
path is **broken at the field-name level** and will raise `ValueError` at runtime.

**Recommendation**: Either (a) add the missing fields to the Rental model, or
(b) update the service to use the correct existing field names.

### 6. Battery.rental_price_per_day / damage_deposit_amount

`RentalService.calculate_price()` accesses `battery.rental_price_per_day` and
`battery.damage_deposit_amount`, which don't exist on the `Battery` model.
These likely belong on a pricing table or the `BatteryCatalog` model.

### 7. No State Machine Enforcement in apply_battery_transition

`apply_battery_transition()` validates the *target* status but not the *from*
status. There's no guard preventing `deployed тЖТ deployed` (double-deploy) or
`retired тЖТ deployed` (except the explicit retired check). A state machine
would be safer.
